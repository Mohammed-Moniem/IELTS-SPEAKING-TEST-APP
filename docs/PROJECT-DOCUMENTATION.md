# Project Documentation — IELTS Speaking Test App (Spokio)

**Last updated:** 2026-01-02

This document is a consolidated “source of truth” for this repository: what it contains, how it runs, how the mobile app talks to the backend, what APIs exist, and what is currently missing or inconsistent.

---

## 1) What this repo is

This repository is a multi-part system:

1. **Mobile app**: `mobile/`

   - Expo + React Native application named **Spokio**.
   - Provides IELTS speaking practice, AI feedback, audio recording, results history, and a social layer (friends, 1:1 chat, group chat, leaderboards, achievements, referrals, QR codes).

2. **Backend API**: `micro-service-boilerplate-main/`

   - Node.js + TypeScript Express server using `routing-controllers`.
   - MongoDB via Mongoose.
   - Socket.IO for real-time chat/presence/typing/read receipts.
   - Integrations:
     - OpenAI (transcription / evaluation / question generation)
     - ElevenLabs (text-to-speech)
     - Stripe (subscriptions)
     - Optional S3 storage for files
     - Optional RabbitMQ

3. **Docs**: `docs/`

   - Many “phase” and “completion” documents. They are useful history, but some are out of date/inconsistent with the current code (see the “Gaps / missing” section).

4. **Other folders**
   - `ielts-speaking-test-main/` currently contains a generic template README (appears to be leftover/placeholder rather than the active app).
   - `ios/` exists at repo root (likely an artifact of native builds or previous structure).

## 2) How everything fits together (system architecture)

### 2.1 Runtime components

- **Mobile** (Expo / RN) runs on:

  - Expo Go in development, or a native build.
  - Uses HTTP (Axios) for REST API.
  - Uses Socket.IO client for real-time.

- **Backend** runs on:

  - Local machine (port **4000** by default) OR exposed through ngrok/LocalTunnel.
  - Serves REST at base path: `http(s)://<host>:4000/api/v1`.
  - Serves Socket.IO at base host: `http(s)://<host>:4000`.

- **MongoDB**
  - Local MongoDB (`mongodb://127.0.0.1:27017/ielts-speaking`) by default.
  - Production connection via `MONGO_URL`.

### 2.2 High-level data flow examples

#### A) Login

1. Mobile calls `POST /api/v1/auth/login`.
2. Backend returns `accessToken`, `refreshToken`, and user payload.
3. Mobile stores tokens using the auth storage helper and schedules token refresh.
4. Mobile connects Socket.IO using the JWT access token.

#### B) Voice evaluation

1. Mobile records audio (or uses input text depending on the screen).
2. Mobile may upload audio (depending on mode) and/or call evaluate endpoint.
3. Backend calls OpenAI for transcription/evaluation and returns structured results.
4. Mobile stores results locally and/or fetches analytics/history from backend.

#### C) Chat message send

1. Mobile emits Socket.IO event `message:send` (or calls REST endpoints for history/search/uploads).
2. Backend stores message in MongoDB and emits `message:receive` to the recipient’s room.
3. If recipient is offline, backend enqueues a push notification.

## 4) Development workflow

### 4.1 One-command local runs

From repo root:

- **Local network** (device + computer same Wi‑Fi):

  - `./start-backend-and-mobile-local.sh`

- **ngrok tunnel**:

  - `./start-backend-and-mobile.sh`

- **LocalTunnel stable URL**:
  - `./start-backend-localtunnel.sh`

What these scripts do:

- Ensure backend port 4000 isn’t in use (kills conflicting processes).
- Start backend using `npx nps serve` in `micro-service-boilerplate-main/`.
- Generate `mobile/.env`:
  - `EXPO_PUBLIC_API_URL` → `.../api/v1`
  - `EXPO_PUBLIC_SOCKET_URL` → base host
  - `EXPO_PUBLIC_API_KEY` → currently set to `local-dev-api-key`
- Start Expo (`npx expo start`) in the desired mode.

### 4.2 Mobile — typical commands

From `mobile/`:

- `npm install`
- `npm run start`
- `npm run ios`
- `npm run android`

### 4.3 Backend — typical commands

From `micro-service-boilerplate-main/`:

- `npm install`
- Dev server: `npx nps serve`
- Tests: `npm test` (Jest)
- Lint: `npm run lint`

There are also DB helper scripts, seeding scripts, and test database utilities.

## 5) Mobile app documentation (deep dive)

### 5.1 App bootstrap (`mobile/App.tsx`)

The root app composes several providers and global services:

- `QueryClientProvider` (TanStack Query)
- `AuthProvider` (JWT session management)
- `ThemeProvider`
- `PointsProvider` (only mounted when authenticated)
- `ErrorBoundary`
- `AudioCacheInitializer`
- Notification initialization + listeners
- Socket.IO connect/disconnect based on auth state

### 5.2 Configuration and environments

Config sources:

1. `mobile/.env` (generated by scripts)
2. `mobile/app.config.js` `extra` block
3. `mobile/src/config.ts` fallbacks

Resolved values:

- `API_BASE_URL` default: `http://localhost:4000/api/v1`
- `SOCKET_URL` default: `http://localhost:4000`
- `API_KEY` default: `local-dev-api-key`

Important security note:

- Any `EXPO_PUBLIC_*` value is bundled into the app; do not put real secrets there.

### 5.3 Navigation structure

The primary navigation is a bottom tab navigator defined in `mobile/src/navigation/AppNavigator.tsx`.

Visible tabs:

- **Practice** → `PracticeNavigator` (stack)
- **VoiceTest** → Voice AI screen
- **Home**
- **Results**
- **Social** → `SocialNavigator` (stack)

Hidden routes accessible via menu/navigation:

- Profile
- Analytics
- Settings
- Subscription
- Usage
- Simulations
- Points detail
- Redeem discount

Feature stacks:

- `PracticeNavigator`:

  - PracticeHome
  - PracticeSession
  - PracticeResultDetail

- `SocialNavigator` includes:
  - Friends list / requests / search
  - User profile
  - Conversations
  - Chat (1:1)
  - Study groups + group chat
  - Leaderboard
  - Achievements
  - Referrals
  - QR code + scanner

### 5.4 Authentication

Auth lives in `mobile/src/auth/AuthContext.tsx`.

Key behaviors:

- Stores `accessToken` + `refreshToken` and user profile.
- Schedules an auto-refresh ~1 minute before token expiry (based on JWT `exp`).
- Integrates with Axios via `attachAuthHandlers(...)` so that API calls:
  - include `Authorization: Bearer <accessToken>`
  - retry once on `401` by calling `POST /auth/refresh`
- Supports “guest” sessions via `EXPO_PUBLIC_GUEST_EMAIL` / `EXPO_PUBLIC_GUEST_PASSWORD`.

### 5.5 Networking (Axios client)

Implemented in `mobile/src/api/client.ts`:

- Base URL: `API_BASE_URL`.
- Header `x-api-key` is attached from `API_KEY`.
- Request logging in dev.
- Timeout: 120 seconds (to allow long AI/TTS operations).
- Special FormData handling to avoid broken boundaries in React Native.

### 5.6 Realtime (Socket.IO)

Mobile connects via `mobile/src/services/socketService.ts` (and hooks like `useSocket`).

Backend expects JWT via:

- `socket.handshake.auth.token` OR
- `Authorization: Bearer <token>` header

### 5.7 Notifications

Mobile uses `expo-notifications` and a service wrapper (`mobile/src/services/notificationService.ts`).

- Initializes on app start.
- Registers listeners for “received” and “tapped”.
- Syncs push token with backend after login.

### 5.8 Offline support

There are dedicated modules for offline storage/caching (e.g. `mobile/src/services/offlineStorage.ts`) and audio caching (`audioCacheService`, `AudioCacheInitializer`).

### 5.9 Analytics + Monitoring

- `sentry-expo` via `monitoringService`.
- Firebase analytics via `firebaseAnalyticsService`.

## 6) Backend documentation (deep dive)

### 6.1 Server boot sequence

`micro-service-boilerplate-main/src/app.ts`:

1. Loads `.env`.
2. Initializes DI (TypeDI) + telemetry loader.
3. Connects MongoDB (and GridFS if configured).
4. Initializes Winston logging.
5. Optionally connects RabbitMQ.
6. Initializes notification scheduler.
7. Starts Express server (via loader).

### 6.2 Environment variables

Backend env template: `micro-service-boilerplate-main/.env.example`.

Important categories:

- App:
  - `APP_PORT` (default 4000)
  - `APP_ROUTE_PREFIX` (default `/api/v1`)
- Mongo:
  - `MONGO_URL`
- JWT:
  - `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, expiries
- OpenAI:
  - `OPENAI_API_KEY`, model, max tokens, temperature
- ElevenLabs:
  - `ELEVENLABS_API_KEY`, voice/model tuning
- Storage:
  - `STORAGE_PROVIDER` (`mongodb` or `s3`)
  - S3 credentials/buckets
- Stripe:
  - `STRIPE_SECRET_KEY`, price ids, webhook secret
- Telemetry:
  - `TELEMETRY_ENABLED`, metrics path/port

### 6.3 Request/response conventions

The backend uses standardized response envelopes (see the boilerplate docs and response helpers in `src/api/responses/`).

### 6.4 Authentication

- Public REST paths include:
  - `/auth/register`, `/auth/login`, `/auth/refresh`, `/health`
- Auth middleware requires `Authorization: Bearer <JWT>`.
- JWT is verified and user is attached as `req.currentUser`.

**Unique-Reference-Code (URC):**

There are middleware utilities that validate and reflect a `Unique-Reference-Code` header (used for request tracing). Some codepaths expect it.

### 6.5 Socket.IO

Implemented in `src/loaders/SocketIOLoader.ts`.

Key features:

- JWT auth middleware (verifies token).
- Personal rooms: `user:<userId>`.
- Group rooms: `group:<groupId>`.
- Tracks multiple devices per user.
- Updates online status (`UserStatusModel`) and notifies friends.

Events handled:

- Direct:
  - `message:send` → emits `message:receive` + `message:sent`
  - `message:read` → emits `message:read:confirm`
- Group:
  - `group:message:send` → emits `group:message:receive`
  - `group:join`, `group:leave`
- Typing:
  - `typing:start`, `typing:stop` → emits `typing:indicator`
- Presence:
  - `status:check` → emits `status:response`
  - emits `user:online` / `user:offline` to friends

## 8) Security, privacy, and operational concerns

### 8.1 Mobile secrets

- Anything in `EXPO_PUBLIC_*` is visible to clients.
- Guest credentials (if used) are also embedded.

### 8.2 Backend protections

`expressLoader` applies:

- Helmet headers
- XSS sanitization of body/query/params
- `express-mongo-sanitize`
- `hpp` for parameter pollution
- request body trimming
- rate limiting (configurable)

### 8.3 Chat encryption

Backend has an `EncryptionService` and env fields describing encryption. Ensure production encryption keys are properly set and rotated.

## 9) What’s missing / gaps / inconsistencies (actionable)

This section is intentionally blunt: it lists the most important missing pieces and mismatches between code and docs.

### 9.1 Documentation gaps

- **No single authoritative doc previously**: the repo had many phase documents, but not a unified “how to run + how it works” guide. This file fixes that, but it should be kept maintained.
- **Outdated architecture note**: [docs/ARCHITECTURE-OVERVIEW.md](docs/ARCHITECTURE-OVERVIEW.md) references **TypeORM + PostgreSQL**; the backend actually uses **Mongoose + MongoDB**.
- **Template project confusion**: `ielts-speaking-test-main/README.md` is a generic template README and does not document this app.

### 9.2 Config/implementation mismatches

- **Mobile sends `x-api-key` on every request** (see `mobile/src/api/client.ts`), but the backend does not appear to validate it anywhere. Either:

  - implement API-key validation on backend, or
  - remove the header from mobile to reduce false sense of security.

- **Dockerfile port mismatch**: backend `.env.example` and start scripts assume **4000**, but `micro-service-boilerplate-main/Dockerfile` exposes **8086**. This is likely wrong for deployment.

### 9.3 CI/CD and quality gates

- **No GitHub Actions workflows** are present in `.github/workflows/`.
  - Missing automated: lint, tests, typecheck, build verification.

### 9.4 Deployment & release

- Backend has a Dockerfile but lacks:
  - documented runtime port and healthcheck command
  - documented production environment setup (Mongo, storage provider, Stripe webhooks, OpenAI/ElevenLabs keys)
- Mobile has no documented EAS build/release pipeline (build profiles, signing, store submission).

### 9.5 API documentation

- Swagger is mentioned in `.env.example` (and appears supported), but there is no “how to access swagger” section or a published OpenAPI spec.

### 9.6 Data & migrations

- MongoDB doesn’t have a typical migration pipeline; changes rely on code discipline.
  - Missing: a clear schema-change process, seed strategy, and backward compatibility notes.

### 9.7 Observability

- Mobile has Sentry + Firebase analytics wrappers.
- Backend has telemetry toggles, but:
  - no documented exporter config
  - no dashboards/runbooks
  - no centralized log format guide

### 9.8 Security hardening

- Several env defaults are “dev” values. For production you still need:
  - strong JWT secrets
  - correct CORS origin restrictions
  - rate-limit tuning
  - Stripe webhook signature verification confirmed end-to-end
  - a secrets-management approach
