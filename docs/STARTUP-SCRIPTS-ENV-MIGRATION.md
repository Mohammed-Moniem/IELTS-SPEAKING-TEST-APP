# Startup Scripts Update - Environment Variables Migration

## Summary

Successfully migrated all three startup scripts from modifying `app.json` directly to using `.env` files for environment configuration.

## Changes Made

### 1. Updated Scripts

#### `start-backend-and-mobile-local.sh`

**Before:** Used `sed` to modify `app.json` with local IP
**After:** Creates/updates `.env` file with local IP using `cat` and heredoc

#### `start-backend-and-mobile.sh`

**Before:** Used Python script to parse and modify `app.json` with ngrok URL
**After:** Creates/updates `.env` file with ngrok URL using `cat` and heredoc

#### `start-backend-localtunnel.sh`

**Before:** Used `sed` to modify `app.json` with LocalTunnel URL
**After:** Creates/updates `.env` file with LocalTunnel URL using `cat` and heredoc

### 2. Configuration Flow

```
Startup Script
    ↓
Writes to mobile/.env
    ↓
Read by app.config.js (process.env.EXPO_PUBLIC_*)
    ↓
Used by Expo configuration
    ↓
Accessed in app via Constants.expoConfig.extra
```

### 3. Files Modified

- ✅ `start-backend-and-mobile-local.sh` - Now writes to `.env`
- ✅ `start-backend-and-mobile.sh` - Now writes to `.env`
- ✅ `start-backend-localtunnel.sh` - Now writes to `.env`
- ✅ `mobile/app.config.js` - Already configured to read from `.env`
- ✅ `mobile/.env` - Auto-generated by scripts
- ✅ `mobile/.env.example` - Template for manual configuration
- ✅ `mobile/.gitignore` - Already ignoring `.env`

## Benefits

### Before (app.json modification)

❌ Scripts had to parse JSON (complex)
❌ Required `sed` (macOS/Linux differences) or Python
❌ Risk of breaking JSON syntax
❌ Harder to debug
❌ Manual edits would be overwritten

### After (.env approach)

✅ Simple `cat` command to write file
✅ Works identically on all platforms
✅ Can't break syntax (plain text)
✅ Easy to debug (just `cat .env`)
✅ Developers can manually edit if needed
✅ Standard practice in industry
✅ Ready for production deployment

## Testing Checklist

To verify everything works:

1. **Test Local IP Script:**

   ```bash
   ./start-backend-and-mobile-local.sh
   ```

   - Check `mobile/.env` contains your local IP
   - App should connect to backend on same WiFi

2. **Test Ngrok Script:**

   ```bash
   ./start-backend-and-mobile.sh
   ```

   - Check `mobile/.env` contains ngrok URL
   - App should connect via ngrok tunnel

3. **Test LocalTunnel Script:**

   ```bash
   ./start-backend-localtunnel.sh
   ```

   - Check `mobile/.env` contains LocalTunnel URL
   - App should connect via LocalTunnel

4. **Verify Manual Override:**
   ```bash
   cd mobile
   echo "EXPO_PUBLIC_API_URL=http://custom-url.com/api/v1" > .env
   npx expo start --clear
   ```
   - App should use custom URL

## Production Readiness

When deploying to production:

1. **Remove Development Scripts**

   - Scripts are for development only
   - Production will use static environment variables

2. **Set Production Environment Variables**

   - In your hosting platform (Vercel, Netlify, AWS, etc.)
   - Example:
     ```
     EXPO_PUBLIC_API_URL=https://api.production.com/api/v1
     EXPO_PUBLIC_SOCKET_URL=https://api.production.com
     EXPO_PUBLIC_API_KEY=secure-production-key
     ```

3. **Build Process**
   - `app.config.js` will read production env vars during build
   - No code changes needed

## Rollback Plan

If issues occur, you can temporarily rollback:

1. Revert the three startup scripts to their previous versions
2. Scripts will write to `app.json` again
3. `app.config.js` has fallback to `Constants.expoConfig.extra.apiUrl`
4. Both approaches can coexist temporarily

However, the new approach is better and should be kept.
